# トレラン大会 傷病者搬送ルート判断支援ツール 要件定義書

## 1. プロジェクト概要

### 目的

トレイルランニング大会において、山中で傷病者が発生した際に、安全かつ迅速な下山ルートを選定するための意思決定支援ツール。事前に登録したコース・エスケープルート情報をもとに、傷病者の位置を入力するとリアルタイムで下山口候補とその比較情報（距離・高低差・高低図）を提示し、救護スタッフが最適なルートを判断できるようにする。

### 利用フロー

#### 起動時

アプリ起動直後にスタート画面が表示され、以下の2択を選ぶ：

- **新規作成**：メインコースのGPXファイルを読み込み、編集モードで起動
- **開く**：既存のZIPファイルを読み込み、編集モードで起動

#### 編集モード（準備段階・大会前）

GPXを読み込んだ状態から、必要な情報を順次追加していく：

1. トレイル／ロード区間の設定（区間境界・区間名）
1. ポイントの登録（エイド・下山口・ヘリポート・駐車場・危険箇所・通行止め・鍵・水場・自販機・食事・小屋・トイレ・カスタム）
1. エスケープルートのGPX追加 or 手描き・分岐点設定
1. 車両アクセスルートの追加（GPX or 手描き）
1. ZIPファイルとして保存（エクスポート）

> 誤操作は「↩ 元に戻す」ボタンで最大30ステップまで取り消せる。

#### 確認モード（大会当日）

編集モードから切り替え、または起動時にZIPを読み込むと確認モードで使用できる：

1. 地図上に全ルート・ポイントが表示された状態で待機
1. 下山口・ヘリポートのオン／オフを必要に応じて切り替え
1. 傷病者発生時：位置を地図タップ／座標入力で指定
1. 下山口候補が**リアルタイムで**一覧表示される（距離・高低差）
1. 「📈 高低図」ボタンで全候補の高低図を重ねて比較できる
1. スタッフが候補を比較して搬送ルートを決定する

> 編集モード⇔確認モードはいつでも切り替え可能。確認モードでは編集操作（ポイント追加・ルート変更）は無効化される。

### 対象ユーザー

- 大会本部スタッフ（PC・タブレット使用）
- 現場救護スタッフ（スマートフォン使用）

### アクセス方法

- ログイン不要（URLを知っていれば誰でもアクセス可能）
- 大会ごとに専用URLを発行する運用を想定

## 2. ルート構造の設計

### ルートの考え方

メインコースは**トレイル区間とロード区間が交互に繰り返す**構造を持つ。下山口はトレイル区間ごとに管理し、ロード区間には下山口を設定しない。

```
       下山口S
            ↑ 戻る
〈トレイル〉──A────B──〈ロード〉──C────D──〈トレイル〉──E──→ 下山口F
                 ↓                              ↓
             エスケープ1                    エスケープ2
                 ↓                              ↓
              下山口X                         下山口Y
```

傷病者がロード区間にいる場合は、ロード区間に下山口が存在しないため、隣接するトレイル区間の下山口のみが候補となる。

### 候補ルートの計算方針

傷病者の位置が特定されると、以下のすべての候補を自動計算して提示する：

|候補パターン        |説明                         |
|--------------|---------------------------|
|メインコース進行方向    |傷病者位置からコースを進んで次の下山口へ       |
|メインコース引き返し    |傷病者位置からコースを戻って手前の下山口へ      |
|エスケープルート（進行方向）|傷病者位置より先の分岐からエスケープして下山口へ   |
|エスケープルート（引き返し）|傷病者位置より手前の分岐に戻ってエスケープして下山口へ|

各候補について、傷病者位置から分岐点までの距離＋エスケープルートの距離を合算して計算する。

### 分岐点の扱い

- エスケープルートはメインコース上の「分岐点座標」と紐づける
- 分岐点はGPXアップロード後に地図上で手動指定する

## 3. 主要機能

### 3-1. スタート画面・モード管理機能

**スタート画面：**

- アプリ起動時に表示される
- 「新規作成」：メインコースGPXファイルを選択して読み込み、編集モードで起動
- 「開く」：ZIPファイルを選択して読み込み、編集モードで起動

**モード切替：**

- メイン画面のヘッダーにモード切替ボタンを常時表示
- 編集モード→確認モード：確認ダイアログを表示して切り替え
- 確認モード→編集モード：いつでも切り替え可能
- 確認モード中は編集操作（ポイント追加・ルート変更・区間設定）をすべて無効化
- 現在のモードをヘッダーに常時表示する

### 3-2. ルートデータ管理機能

- GPXファイルのアップロードによるルート取り込み
- 地図上をクリックして頂点を打ち、ポリラインを手描きする方式でもルートを追加できる（エスケープ・車道どちらも対応）
- ルートごとの属性設定：
  - ルート名（ダブルクリックで編集ダイアログを開いて変更可能）
  - 種別（`course`：メインコース ／ `escape`：エスケープルート ／ `road_access`：車道）
  - 難易度・危険度（低／中／高）
  - 搬送手段の適性（歩行可／担架可／ヘリ対応）
- エスケープルートの場合：メインコース上の分岐点を地図上で指定

**車両アクセスルートの登録（参考表示用）：**

- 下山口・ヘリポート・駐車場に到着した後の車道ルートを地図上に表示するための機能
- ルート計算には使用しない（スタッフへの視覚的な案内が目的）
- 追加方法：手描き or GPXアップロード
- 地図上では青灰色の点線スタイルで表示
- 種別：`road_access`として管理

**トレイル／ロード区間の設定（メインコースのみ）：**

- GPXアップロード後、地図上でルートを区間分けする
- 各区間に種別（`trail`／`road`）を設定する
- トレイル区間・ロード区間ともに任意で**区間名**をつけられる
- 設定していない区間は暗黙的にトレイルとして扱われ、リストに表示される
- 区間の境界点はGPXの最近傍座標に丸めて確定する
- トレイル区間と隣接するロード区間の境界点には**下山口（自動）**が自動生成される（重複チェックあり）
- 区間クリックで地図上の該当区間を表示、ダブルクリックで名前編集ダイアログを表示
- 区間ごとに「編集」「🗑」ボタンを表示
- 区間情報はrace.jsonに保存される（GPXファイル自体は変更しない）

**地図表示色：**

|要素|色|スタイル|
|---|---|---|
|トレイル|赤 `#dc2626`|実線|
|ロード|青 `#2563eb`|点線|
|エスケープルート|青 `#2563eb`|実線|
|車道|青灰色 `#9ca3af`|点線|
|ルート候補|オレンジ・紫・シアンなど（候補ごとに色分け）|実線・太め|

### 3-3. ポイント管理機能

大会ごとに地図上のポイントを登録・管理する。

**ポイントの種別：**

|種別|アイコン|ルート計算|説明|
|---|---|---|---|
|下山口|🚩|ゴール地点|徒歩搬送の終着点|
|ヘリポート|🚁|ゴール地点|徒歩でそこに辿り着くことをゴールとしてルート計算の対象となる|
|エイド|🏕️|対象外|給水・補給ポイント（参考表示のみ）|
|駐車場|🅿️|対象外|車両集結ポイント（参考表示のみ）|
|危険箇所|⚠️|対象外|滑落・転落などの危険箇所|
|通行止め|🚫|対象外|通行不可区間の入口|
|鍵|🔑|対象外|施錠されたゲートなど|
|水場|💧|対象外|天然の水場・給水ポイント|
|自販機|🥤|対象外|自動販売機|
|食事|🍽️|対象外|食事提供箇所|
|小屋|🛖|対象外|山小屋・避難小屋|
|トイレ|🚻|対象外|トイレ|
|カスタム|📍|対象外|任意の名前をつけて自由に登録|

**ルート計算ゴール地点（下山口・ヘリポート）のオン／オフ機能：**

- 下山口・ヘリポートは候補に含めるかどうかをチェックボックスでオン／オフできる
- **準備フェーズ・運用フェーズ両方で切り替え可能**（当日の状況変化に対応）
- オフにしたポイントは地図上でグレー表示され、ルート計算の対象外となる

**ポイントの追加方法：**

- 地図上をクリック／タップして位置を指定し、種別・名前・備考を入力して登録
- 緯度・経度を手入力して登録
- **ルート近傍スナップ**：クリック位置がいずれかのルートの100m以内にある場合、「ルート上にポイントを追加しますか？」の確認ダイアログを表示する
  - 「ルート上に追加」：ルート上の最近傍点にスナップする。垂足がGPX頂点から5%以上離れている場合は新しい座標をGPXに挿入し、標高もAPIから取得する
  - 「そのまま追加」：クリック位置そのままで追加する

**ポイントの操作：**

- サイドパネルのポイント一覧をクリック → 地図上でそのポイントに移動
- サイドパネルのポイント一覧をダブルクリック（編集モード）→ 編集ダイアログを開く
- 編集ダイアログ：種別・名前・備考・写真を編集できる
- 編集ボタン・削除ボタンを各ポイント行に表示

**写真の登録：**

- ポイントごとに複数枚の写真を添付できる（data URLとして保存）

### 3-4. 地図表示機能

- 国土地理院地図をベースマップとして使用
- コース・エスケープルート・車道をオーバーレイ表示（種別・区間種別で色分け）
- 登録ポイントを種別ごとに異なるアイコンで表示
- 傷病者位置のマーカー表示（🚨）
- ルート候補選択時：該当ルートをハイライト表示
- **候補表示中のディム**：傷病者位置を指定してルート候補が表示されている間は、候補ルート以外のルートとポイントを半透明（opacity 0.3程度）で表示
- サイドパネルの各項目クリックで地図ビューを移動：
  - ルート → ルート全体にfitBounds
  - 区間 → 区間全体にfitBounds
  - ポイント・ゴール地点 → そのポイントにpanTo

### 3-5. データ入出力機能（ZIPファイル）

大会データをZIPファイル単位でインポート・エクスポートできる。

**ZIPファイル構成：**

```
race_xxtrail_2026.zip
  ├── race.json              # 大会情報・ポイント・ルート定義・分岐点情報
  ├── course_main.gpx        # メインコースのGPX
  ├── escape_1.gpx           # エスケープルート1のGPX
  ├── escape_2.gpx           # エスケープルート2のGPX
  └── road_access_1.gpx      # 車両アクセスルートのGPX（任意）
```

**エクスポート：**

- 準備フェーズ完了後、現在の大会データをZIPで一括ダウンロード

**インポート：**

- ZIPファイルをアップロードするだけで大会データを復元
- 運用フェーズ開始時にこの操作を行う

### 3-6. 傷病者位置入力機能

以下の複数の方法で位置を指定可能：

- 地図上をタップ／クリックして指定
- GPS座標（緯度・経度）を手入力

### 3-7. ルート候補計算・表示機能（リアルタイム）

傷病者位置が指定されると即座に計算を実行し、すべての下山口候補を一覧表示する。

**表示項目：**

|項目|説明|
|---|---|
|下山口名|ゴール地点の名前|
|ルート概要|例：「メインコース進行方向」「エスケープ1経由（前方分岐）」|
|距離|傷病者位置から下山口までの合計距離（km）|
|累積標高差（下り）|下りの累積標高差（m）|
|累積標高差（上り）|引き返し時など上りが発生する場合（m）|
|難易度・危険度|低／中／高|
|搬送手段の適性|歩行可／担架可／ヘリ対応|

**高低図比較：**

- 「▼ ルート候補」セクションヘッダー横の「📈 高低図」ボタンを押すと、全候補を1つのSVGチャートに重ね描きした比較モーダルを表示
- X軸：実距離（0〜各候補の総距離）
- Y軸：全候補共通の標高範囲
- 各候補ラインは CANDIDATE_COLORS で色分け
- 傷病者（🚨）・ゴール（🚩/🚁）・ルート上のポイントをクリッカブルマーカーで表示（クリックで名前・備考ポップアップ）
- 凡例に色・名前・ラベル・距離・下り・上りを表示

**候補に含める下山口の条件：**

- 下山口のオン／オフが「オン」のもののみ計算対象とする
- 傷病者がロード区間にいる場合、そのロード区間には下山口候補を生成しない（隣接トレイル区間の下山口のみ対象）
- スナップ距離が100mを超えるルートは候補から除外

**UI：**

- カード形式で複数候補を並列表示
- 距離の短い順にデフォルト表示
- カードを選択すると地図上に該当ルートをハイライト（他はディム）

### 3-8. 編集操作の取り消し（Undo）

- 編集パネル上部の「↩ 元に戻す」ボタンで直前の操作を取り消せる
- 最大30ステップまでスタック（古いものから自動削除）
- 対象操作：ポイント追加・更新・削除、ルート追加・更新・削除、区間設定・分岐点設定、大会名変更
- GPX/ZIPの読み込み時は履歴をリセット

### ルート候補計算結果（フロントエンド内部）

```typescript
type RouteCandidate = {
  id: string
  label: string            // 例：「エスケープ1経由（前方分岐）」
  exitPointId: string
  exitPointName: string    // ゴール地点名
  exitPointType: 'exit' | 'helipad'
  segments: CandidateSegment[]
  totalDistanceM: number
  totalDescentM: number
  totalAscentM: number
  difficulty: 'low' | 'medium' | 'high'
  transportSuitability: ('walk' | 'stretcher' | 'helicopter')[]
  pathCoords: LatLngEle[]  // 高低図描画用の座標列
}

type CandidateSegment = {
  routeId: string
  direction: 'forward' | 'backward'
  fromIndex: number
  toIndex: number
}
```

## 6. 技術要件

### 構成方針

- **フロントエンドのみで完結**（バックエンドサーバー不要）
- データの永続化はZIPファイルのエクスポート／インポートで代替
- ルート計算はすべてブラウザ上のJavaScriptで処理
- GitHub Pagesに静的サイトとしてデプロイ

### 技術スタック

|項目|採用技術|備考|
|---|---|---|
|フレームワーク|React + TypeScript|コンポーネント管理・型安全性|
|ビルドツール|Vite|高速ビルド|
|地図|Leaflet.js + react-leaflet|国土地理院タイル対応|
|GPXパース|gpxparser|シンプルで軽量|
|ZIP処理|JSZip|ブラウザでZIP読み書き可能|
|状態管理|Zustand|軽量・シンプル|
|スタイル|Tailwind CSS v4|レスポンシブ対応が速い|
|ホスティング|GitHub Pages|無料・GitHub連携|

### プロジェクト構成

```
src/
  components/
    map/
      MapView.tsx           # メイン地図（ルート・ポイント描画、コマンド受信）
```

## 10. ルート計算ロジック設計

### 概要

傷病者の任意座標を受け取り、登録済みの全ルート・全ゴール地点（下山口・ヘリポート）の組み合わせに対してリアルタイムで距離・標高差を計算し、候補一覧を返す。

-----

### ステップ1：最近傍点（スナップ点）の特定

傷病者の任意座標Pから、各GPXルートの座標列に対して最近傍点を求める。

```
傷病者位置 P
     ↓ 垂線を下ろす
GPX座標列：P0 ─ P1 ─ P2 ─ [スナップ点N] ─ P3 ─ P4
```

**処理手順：**

1. GPXの各線分（Pi → Pi+1）に対して、Pから線分への最短距離を計算
1. 最短距離が最小の線分を特定し、その垂足を「スナップ点N」とする
1. スナップ点は `segmentIndex + ratio`（例：2.7 = P2とP3の間70%地点）として管理
1. **スナップ距離が100mを超えるルートは候補から除外する**

-----

### ステップ2：ゴール地点までの経路抽出

スナップ点Nから各ゴール地点までのGPX座標列のスライスを取り出す。

**ゴール地点のスナップ：**

- ゴール地点を座標列にスナップする際、境界座標（トレイル↔ロードの接続点）への帰属を正しく処理するため、ratioを最近傍の頂点インデックスに丸める（ratio ≥ 0.5 なら segmentIndex+1 の頂点）
- これにより、境界点がロード側のエッジでなくトレイル側のエッジに帰属し、`hasRoadSection` の誤判定を防ぐ

**メインコース（進行・引き返し）：**

```
進行方向：N → P3 → P4 → ... → ゴールA
戻り方向：N → P2 → P1 → ... → ゴールB（座標列を逆順に辿る）
```

**エスケープルート経由：**

```
（前方分岐）N →前進→ 分岐点J → エスケープGPX Q0 → Q1 → ... → ゴールX
（後方分岐）N →後退→ 分岐点J → エスケープGPX Q0 → Q1 → ... → ゴールX
```

-----

### ステップ3：距離・標高差の計算

抽出した座標列に対して順次計算する。

```typescript
// 距離：Haversine公式で隣接点間の球面距離を合算
totalDistanceM = sum of haversine(Pi, Pi+1)

// 標高差：上り・下りを分けて累積
for each (Pi, Pi+1):
  diff = Pi+1.ele - Pi.ele
  if diff > 0: totalAscentM += diff
  if diff < 0: totalDescentM += Math.abs(diff)
```

-----

### ステップ4：候補一覧の生成

|候補パターン|除外条件|
|---|---|
|メイン進行|前方にゴールがない／スナップ距離100m超|
|メイン引き返し|後方にゴールがない／スナップ距離100m超|
|エスケープ（前方分岐）|前方に分岐点がない|
|エスケープ（後方分岐）|後方に分岐点がない|

**ロード区間の除外：**

セグメントマップ `segMap[i]` = edge `i` の terrain を構築し、候補ルートが通過するエッジ範囲にロードエッジが含まれる場合はその候補を除外する（ロード区間にゴール地点は存在しないという設計方針に基づく）。

**ソート：** 距離の短い順。

## 11. 状態管理設計（Zustand）

### ストア構成

ストアは4つに分割し、関心ごとに状態を管理する。

-----

### raceStore（大会データ）

ZIPまたはGPXから読み込んだデータをすべて保持する。編集操作のたびにスナップショットを`history`に積み、`undo()`で復元できる。

```typescript
type RaceStore = {
  race: Race | null
  routes: Route[]
  points: Point[]
  history: Snapshot[]       // 最大30件

  loadFromGpx: (file: File) => Promise<void>
  loadFromZip: (file: File) => Promise<void>
  exportToZip: () => Promise<void>
  undo: () => void

  setRace: (r: Race) => void
  addRoute: (route: Route) => void
  updateRoute: (id: string, patch: Partial<Route>) => void
  deleteRoute: (id: string) => void
  setSegments: (routeId: string, segments: Segment[]) => void
  setJunction: (routeId: string, junction: Junction) => void

  addPoint: (point: Point) => void
  updatePoint: (id: string, patch: Partial<Point>) => void
  deletePoint: (id: string) => void
  togglePoint: (id: string) => void
}
```

-----

### modeStore（モード・操作状態）

編集／確認モードと、編集中のアクティブな操作ツールを管理する。

```typescript
type Mode = 'edit' | 'operation'

type EditTool =
  | 'none'
  | 'add_point'
  | 'set_segment'
  | 'set_junction'
  | 'draw_route'

type ModeStore = {
  mode: Mode
  activeTool: EditTool
  setMode: (mode: Mode) => void
  setActiveTool: (tool: EditTool) => void
}
```

-----

### casualtyStore（傷病者・計算結果）

```typescript
type CasualtyStore = {
  position: LatLng | null
  candidates: RouteCandidate[]
  selectedCandidateId: string | null

  setPosition: (pos: LatLng, candidates: RouteCandidate[]) => void
  selectCandidate: (id: string | null) => void
  clearCasualty: () => void
}
```

-----

### mapStore（地図コマンド）

パネルから地図のビューを操作するためのコマンドストア。

```typescript
type MapCommand =
  | { seq: number; type: 'fitBounds'; latlngs: LatLng[] }
  | { seq: number; type: 'panTo'; latlng: LatLng }

type MapStore = {
  command: MapCommand | null
  fitBounds: (latlngs: LatLng[]) => void
  panTo: (latlng: LatLng) => void
}
```

MapView がこのストアを購読し、`command` が変化したときに `map.fitBounds()` / `map.setView()` を実行する。`seq` を持つことで同じ座標への連続コマンドも検知できる。

-----

### ZIPの読み書き方針

|タイミング|動作|
|---|---|
|新規作成|GPX読み込み → raceStoreに格納 → 編集 → 手動エクスポート|
|開く|ZIP読み込み → raceStoreに格納 → 編集 → 手動エクスポート|
|保存|エクスポートボタンを押した時だけZIPを生成・ダウンロード|
|自動保存|なし（明示的な操作のみ）|

## 13. 編集操作UX設計

### 区間境界の設定

```
1. 「＋ 区間追加」ボタンを押す
         ↓
2. 地図上のメインコース上をクリックして開始点を指定
   → クリック位置に最も近いGPX座標点に丸める
         ↓
3. 同様に終了点を指定
         ↓
4. ポップアップで種別（トレイル or ロード）を選択して確定
   → 境界点にある場合、下山口（自動）を自動生成
```

### ポイントの追加

```
1. 「＋ ポイントを追加」ボタンを押す
         ↓
2. 地図上の任意の場所をクリック
         ↓
3. 近くにルートがある場合（100m以内）：
   「ルート上にポイントを追加しますか？」ダイアログ
   → 「ルート上に追加」：最近傍点にスナップ（必要に応じてGPX座標を挿入・標高取得）
   → 「そのまま追加」：クリック位置を使用
         ↓
4. ポイント追加ダイアログで種別・名前・備考・写真を設定して「追加」
```

### エスケープルートの追加

```
GPXアップロード or 手描き（✏️ エスケープ(手描き)ボタン）
         ↓
ルートが地図上に表示される
         ↓
「分岐点を設定」ボタンを押してメインコース上の分岐点をクリック
         ↓
分岐点マーカーが確定する
```

### モード切替

```
編集 → 確認：
  「確認モードへ」ボタンを押す
  → 確認ダイアログ表示
  → 「続ける」で確認モードへ切り替え
  → 編集操作がすべて無効化される

確認 → 編集：
  「←編集」ボタンを押す
  → 即座に切り替え（ダイアログなし）
  → 傷病者位置・計算結果はクリアされる
```
